The chunking uses LangChain's RecursiveCharacterTextSplitter with these settings:
chunk_size=512: Maximum 512 characters per chunk
chunk_overlap=50: Adjacent chunks share 50 characters of overlap
separators: ["\n\n", "\n", " ", ""] - tries to split on paragraph breaks first, then line breaks, then spaces, then characters
This means each chunk is roughly 512 characters long (about 100-150 words), with a 50-character overlap to maintain context continuity between chunks.

Yes! The LLM summarization starts with the individual 512-character chunks and processes them hierarchically:

For chapters with MORE than 10 chunks:

Step 1: Processes chunks in batches of 3

Takes up to the first 30 chunks
Extracts up to 1000 characters from each chunk's text
Combines 3 chunks per batch
Sends each batch to LLM to get a 2-3 sentence summary
Step 2: Combines all those batch summaries

Step 3: Sends the combined summaries to LLM for final summary at target word count (100 or 200 words)

For chapters with 10 or FEWER chunks:

Skips batch summarization
Takes up to 800 characters  each of the first 15 chunks
Combines them directly
Sends to LLM for final summary
So the process is:

Individual 512-char chunks → (optional intermediate summaries) → final summary
This hierarchical approach prevents overwhelming the LLM with too much text at once while maintaining the important information from the original chunks


------------
The PDF summary gets all the chapter summaries appended together, not the original chunks.

The hierarchical process is:

Individual 512-char chunks →
Chapter summaries (~100 words each) →
PDF summary (~200 words)
At line 585, all chapter summaries are combined:
Then up to the first 10,000 characters of these combined chapter summaries (line 595) are sent to the LLM with the prompt asking it to synthesize the main themes across chapters into a 200-word overview.

So it's a true hierarchical summarization:

Chunks → Chapter summaries → Book summary
Not: Chunks → Book summary

No, the summarization process does not use the vector embeddings at all. Here's what happens:

Chunks are retrieved from ChromaDB using .get() (line 100) - this retrieves all chunks without any vector similarity search
Only the text is extracted: results['documents'][i] (line 112) - this is just the raw text, not the embeddings
The text is sent to the LLM for summarization
The vector embeddings stored in ChromaDB are not used for the summarization. They would be used if you were doing:
	Semantic search (finding similar chunks)
	Question answering (retrieving relevant chunks based on a query)
	RAG (Retrieval Augmented Generation)

But for summarization, the process simply:
Retrieves all chunks (using .get() not .query())
Extracts the text content only
Sends the text to the LLM
The embeddings are stored but not utilized in the current summarization workflow. They're there if you want to add search/retrieval capabilities later.

what is a .pkl (pickle?) file?

A .pkl file is a Python pickle file - Python's native binary serialization format for saving Python objects to disk.

What it does:

Saves Python objects (dictionaries, lists, custom objects, etc.) to a file
Preserves the exact structure and data types
Much faster to load than parsing text files like JSON

Fine-tuning

What’s Feasible on Your Hardware
Full fine-tuning (updating all model weights) is not practical on 32 GB RAM, even with Apple Silicon. It requires much more memory and is typically done on high-end GPUs.
PEFT/LoRA fine-tuning is practical and recommended. This approach only updates a small number of additional parameters, making it much less memory-intensive and feasible on your Mac.
What You Can Achieve
Domain adaptation: The model will better understand your specific vocabulary, topics, and style.
Task adaptation: You can teach the model to summarize in your preferred format, tone, or for your target audience.
Instruction tuning: You can provide example prompts and desired outputs to guide the model’s behavior.

How to Do It
1. Prepare your data:
Format your corpus as instruction/response pairs or (document, summary) pairs in JSON, CSV, or text.

2. Use HuggingFace Transformers + PEFT:
Install:  pip install transformers peft datasets
Use a LoRA fine-tuning script. HuggingFace provides examples.
Set batch size and sequence length to fit your RAM (e.g., batch_size=1–2, max_seq_length=1024–2048).

3. Run training:
Training will be slow, but it will work.
Save only the LoRA adapter weights (very small files).

4. Inference:
Load the base Qwen2.5-7B-Instruct model and your LoRA adapter for summarization.
Example: LoRA Fine-Tuning Workflow

Tips
Start small: Try a subset of your data first to ensure everything works.
Monitor RAM usage: Use Activity Monitor to avoid swapping.
Use mixed precision (float16): This reduces memory usage.
Training time: Expect hours to days, but it’s feasible.

chapter_cache.pkl (binary, Python pickle)

Contains chapter titles, metrics, and summary fields, but not the full chapter texts.
Used for quick lookups, reporting, and as a reference for what’s in the main cache.

summaries_cache.json: Stores summaries in a numeric-indexed format for fast report generation.
chapter_index.json: Maps book/chapter numbers to chapter names and metadata (no full text).
